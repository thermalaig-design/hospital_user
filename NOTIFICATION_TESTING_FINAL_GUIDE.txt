        Error Code: 42501
        Error Message: permission denied for table notifications
        Error Details: null
        Error Hint: Check your RLS policies
    ❌ Booking notification FAILED for 9999999999: permission denied...


WHAT EACH ERROR CODE MEANS:

    42501 = Permission denied (RLS policy issue)
            Fix: Check Supabase RLS policies on notifications table
            
    23502 = NOT NULL constraint violation
            Fix: One of the required columns is missing data
            
    42703 = Column doesn't exist
            Fix: notifications table schema is wrong
            
    08P01 = Supabase connection failed
            Fix: Check your .env SUPABASE_URL and KEY

🔍 DETAILED TROUBLESHOOTING
─────────────────────────────────────────────────────────────────────────────────

ISSUE 1: Backend crashes after booking appointment
═════════════════════════════════════════════════════════════════════════════════

ERROR: Cannot read property 'success' of undefined

CAUSE: The sendInAppNotification function is returning wrong value
FIX: Make sure latest code is in appointmentController.js line 1-45

VERIFY:
    const result = await sendInAppNotification(...);
    console.log(result); // Should be: { success: true, data: ... }


ISSUE 2: "No notifications yet" in app even though logs say ✅
═════════════════════════════════════════════════════════════════════════════════

CAUSE: Phone number mismatch between:
    - localStorage user.mobile
    - appointments.patient_phone
    - notifications.user_id

DEBUG:
1. Book appointment, get ID
2. Check Supabase:
   SELECT * FROM appointments WHERE id = [ID];
   → Check patient_phone value
   
   SELECT * FROM notifications 
   ORDER BY created_at DESC LIMIT 1;
   → Check user_id value
   
3. Browser console:
   JSON.parse(localStorage.getItem('user')).mobile
   → Should match both above

FIX:
   If doesn't match, update one of these:
   a) appointments table: UPDATE appointments SET patient_phone = '9999999999'
   b) OR login again to reset localStorage


ISSUE 3: All appointment operations work but get "notification: { success: false }"
═════════════════════════════════════════════════════════════════════════════════

CAUSE: Supabase permissions issue

CHECK:
1. Backend logs for full error message
2. Look for "Error Code:", "Error Message:", "Error Hint:"
3. Most common: RLS (Row Level Security) policy blocking insert

FIX (in Supabase):
1. Go to notifications table → RLS Policies
2. Create new policy:
   - Name: "Allow anonymous insert for testing"
   - Command: INSERT
   - Check: (true)  ← This allows anyone to insert
3. Test again

PROPER POLICY (after testing):
   - Check: (auth.uid() = cast(user_id as uuid))
   - This matches only the user's own notifications


ISSUE 4: Booking appointment takes 10+ seconds to respond
═════════════════════════════════════════════════════════════════════════════════

CAUSE: Notification insert is slow (usually network issue)

CHECK:
1. Backend logs - how long does notification insert take?
2. Check Supabase server status
3. Check internet connection speed

SOLUTION:
    Don't wait for notification response in frontend
    (This is a separate feature to add later)
